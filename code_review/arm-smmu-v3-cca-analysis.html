1 SMMUv3 硬件背景

Background
什么是 SMMUv3？

SMMU (System Memory Management Unit) v3 是 ARM 架构的 IOMMU 标准，v3 版本引入了大量从 CPU MMU 借鉴的设计理念。它为 PCIe 和 AMBA 总线上的 DMA 设备提供地址翻译和保护，是虚拟化、机密计算和 DMA 安全的关键组件。

1.1 核心硬件概念

概念说明
Stream (流)一次 DMA 事务，由 Stream ID (SID) 标识，类似 CPU 的 ASID。每个 PCIe 设备 function 分配一个 SID。
STE (Stream Table Entry)流表项，硬件查找 SID 得到 STE，决定翻译策略：bypass、stage-1、stage-2、或嵌套 (nested)。
CD (Context Descriptor)上下文描述符，用于 Stage-1 翻译，包含页表指针、ASID、TCR 等，类似 CPU 的 TTBR0_EL1。
Stage-1 (S1)VA → IPA 翻译，对应 Guest OS 页表，类似 CPU EL1/0 翻译。
Stage-2 (S2)IPA → PA 翻译，对应 Hypervisor 页表，类似 CPU EL2 Stage-2 翻译。
CMD Queue命令队列，软件写入 TLB 刷写 (TLBI)、配置失效 (CFGI)、同步 (SYNC) 等命令。
EVT Queue事件队列，硬件报告翻译故障、配置错误等事件。
PRI QueuePRI (Page Request Interface) 队列，PCIe PRI 协议支持，用于 SVA (Shared Virtual Addressing)。
ECMDQ扩展命令队列，per-CPU 私有命令队列，减少锁竞争 (硬件可选特性)。

1.3 命令同步机制

SMMUv3 通过 CMD_SYNC 命令保证命令队列之前的命令已被硬件消费完毕。三种同步完成检测方式：

Sync Methods
  • SEV — 发送 SEV (Send Event) 唤醒轮询的 CPU，通过轮询 CONS 指针完成，延迟低但占用 CPU
  • MSI — 命令完成后发送 MSI 中断，释放 CPU 资源，延迟较高但适合异步场景
  • Polling — 直接在 CONS 寄存器上轮询，模块参数 disable_msipolling 可禁用 MSI polling

    ARM_SMMU_FEAT_TRANS_S2      // 支持 Stage-2

    ARM_SMMU_FEAT_E2H           // EL2 E2H 模式

    ARM_SMMU_FEAT_SVA           // Shared Virtual Addressing

3.4 关键设计点

Design Highlights
  • STE 写入使用 CMX 比较-交换模式 (compare-and-swap) 保证原子性，SEE arm_smmu_write_entry()
  • ECMDQ (per-CPU 私有 CMDQ) 绕过主 CMDQ 的锁竞争，对性能敏感场景至关重要，需硬件支持 IDR1.ECMDQ
  • 二级流表 (2-level STR) 支持稀疏 SID 空间，L1 表 + L2 表延迟分配，避免线性表的连续大块物理内存需求
  • PRI + IOPF 集成 io-pgfault 框架，支持 SVA 的按需页调度
  • MSI Polling 通过 MSI 中断完成 CMD_SYNC 通知，使用模块参数 disable_msipolling 控制
  • TLBI 范围失效 (Range Invalidation) 利用硬件 IDR3.RIL 特性批量失效，减少命令队列压力

5 arm-r-smmu-v3.c — Realm SMMU (821 行)

Tag
Realm SMMU — ARM CCA 机密计算扩展

基于 CONFIG_HISI_CCADA_HOST 编译，为 ARM CCA (Confidential Compute Architecture) 的 Realm 提供隔离的 SMMU 管理能力。所有硬件访问通过 RMI SMC 调用委托给 RMM (Realm Management Monitor) 固件。

5.1 设计思想

ARM CCA 引入 Realm 概念 — 比 TrustZone 更强的隔离执行环境。Realm 中的 DMA 设备需要 SMMU 翻译，但 Host 内核 (Non-Secure EL2) 不能直接访问 Realm 的 SMMU 寄存器。R-SMMU 的设计模式是：

  • 双 SMMU 实例：每个物理 SMMU 硬件在逻辑上分为 arm_smmu_device (NS SMMU) 和 realm_smmu_device (R-SMMU)，共享同一组物理寄存器地址空间
  • RMI 封装：所有 R-SMMU 寄存器访问通过 rmi_smmu_reg_read32/write32 → arm_smccc_1_2_smc → RMM 完成
  • 命令转发 (Command Forwarding)：非安全侧提交 CMDQ 命令时，筛选出影响 Realm 的命令 (TLBI/CFGI 等)，复制到 R-CMDQ 由 RMM 执行
  • 独立队列：R-CMDQ 和 R-EVTQ 是独立于 NS CMDQ/EVTQ 的队列，通过 granule_delegate 将队列内存委托给 Realm 世界

5.3 命令转发机制

这是 R-SMMU 最精妙的设计。当 NS SMMU 驱动提交命令到 CMDQ 时，realm_smmu_cmdq_issue_cmdlist() 会筛选出需要转发到 R-CMDQ 的命令：

static bool realm_smmu_need_forward(struct arm_smmu_device *smmu, u64 cmd0, u64 cmd1)
{
    // 根据 opcode 判断是否需要转发:
    switch (opcode) {
    case CMDQ_OP_TLBI_EL2_ALL:   // EL2 全局 TLB 失效 → 始终转发
    case CMDQ_OP_TLBI_NSNH_ALL:  // Non-Secure Non-Hyp 全局 → 始终转发
    case CMDQ_OP_CFGI_ALL:       // 全 STE 失效 → 始终转发
        return true;

    case CMDQ_OP_TLBI_S2_IPA:    // S2 IPA 范围内失效
        return is_realm_vmid(smmu, FIELD_GET(CMDQ_TLBI_0_VMID, cmd0));

    case CMDQ_OP_CFGI_STE:       // 特定 STE 失效
        return is_realm_sid(smmu, FIELD_GET(CMDQ_CFGI_0_SID, cmd0));

    // ... ATC_INV, CFGI_CD, RESUME, PRI_RESP 同理
    }
}

// 筛选出需转发的命令 → 打包 → rmi_smmu_send_cmdlist() → RMM
int realm_smmu_cmdq_issue_cmdlist(...)
{
    for (i = 0; i < n; i++) {
        if (realm_smmu_need_forward(smmu, cmd0, cmd1))
            rcmds[forward_cnt++] = ...;
    }
    // 通过 SMC 将筛选后的命令列表发送到 R-CMDQ
    rmi_smmu_send_cmdlist(smmu->realm.ioaddr, rcmds_pa, forward_cnt, sync);
}

6 hisi-cca-da.c — HiSilicon CCA Domain Access (1456 行)

Tag
CCA DA — 设备委派与 DMA 管理

位于 arch/arm64/kvm/hisi-cca-da.c，是 CCA 方案中设备委派 (Device Delegation) 的核心实现。管理 PCIe 设备在非安全世界与 Realm 世界之间的所有权转移。

6.1 设计思想

CCA DA 模块解决的核心问题：如何在 CCA 架构下安全地将物理 PCIe 设备分配给 Realm VM。

关键设计模式：

  • 两级哈希表追踪: g_root_dev_htable (root device → count/delegation) + g_realm_dev_htable (device + vttbr → realm entry)
  • RMI 设备委派: 通过 rmi_dev_delegate/rmi_dev_attach 等 SMC 调用将设备所有权转给 RMM
  • MMIO 访问拦截: 对已委派的 Realm 设备，readl/writel 等操作被挂钩到 RMM (通过 rmi_dev_mmio_read/write)
  • PCI 配置空间保护: ccada_pci_generic_config_read/write 将 Config 访问转发到 RMM 保护
  • MSI/MSI-X 重映射: Realm 设备的 MSI IOVA 被重新映射，通过 rme_get_msi_iova() 在 RMM 分配的页面上设置

6.5 MMIO 访问钩子

委派设备的 MMIO 访问通过 RMI 转发到 RMM 执行，确保 Non-Secure 世界无法访问 Realm 设备的 MMIO 空间：

// 对 delegated 设备的 readl/writel 钩子
u32 rme_readl_hook(void __iomem *addr, struct pci_dev *pdev)
{
    if (is_support_rme() && is_dev_delegated(pdev))
        // 使用 AT S1E1W 将 VA 转为 PA
        // 然后通过 rmi_dev_mmio_read() SMC 调用让 RMM 读取
        return rme_mmio_read32(rme_mmio_va_to_pa(addr), pdev);

    return readl(addr); // 普通设备走正常路径
}

支持的钩子全家桶：rme_readl_hook, rme_writel_hook, rme_read32be_hook, rme_write32be_hook, rme_lo_hi_writeq_hook, rme_hi_lo_writeq_hook, rme_lo_hi_readq_hook, rme_read8_hook, rme_writeq_hook, __rme_raw_writel_hook, __rme_iowrite64_copy_hook。
