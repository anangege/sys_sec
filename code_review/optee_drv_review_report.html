严重问题
CR-01CVSS 9.1
hpre_ecc.c:860-902 — gen_random_k()

ECDSA 随机数 k 的高位存在严重偏置

ECDSA 随机数 k 的高字节从曲线阶 n 确定性地拷贝生成，仅有低字节是随机的：

for (i = 0; i < size; i++) {
    if (n[i] > 1) {
        rand_k[i] = n[i] - 1;  // 高位固定为 (n_hi - 1)
        break;
    }
    rand_k[i] = n[i];
}
j = i + 1;
hw_get_random_bytes(rand_k + j, size - j);  // 仅低字节随机

熵严重降低：

  • P-256: k = 0xFE || 随机(31字节) → 248 位（丢失 8 位）
  • P-384: k = 0xFE || 随机(47字节) → 376 位（丢失 8 位）
  • P-521: k = 0x01 0xFE || 随机(64字节) → 512 位（丢失 16 位）
  • SM2: 模式同 P-256

影响: ECDSA 随机数偏置可通过格攻击（lattice attack）从数千个签名中恢复私钥，类似于 2010 年 PlayStation 3 ECDSA 私钥泄露事件。

修复建议

将 gen_random_k() 重写为在 [1, n-1] 范围内均匀生成随机数：

do {
    hw_get_random_bytes(rand_k, size);
    // 对非整字节曲线（如 P-521）掩码高位
    rand_k[0] &= mask;
} while (is_zero(rand_k, size) || is_gte(rand_k, n, size));
CR-02CVSS 8.7
hisi_qm.c:243-259 — hisi_qm_mb_read()

hisi_qm_mb_read() 丢弃硬件信箱数据

信箱读取函数错误地使用 *base（调用者传入的零值）覆写了硬件读取后的数据结构：

static enum hisi_drv_status hisi_qm_mb_read(struct hisi_qm *qm, uint64_t *base, ...)
{
    qm_mb_init(&mb, cmd, 0, qnum, QM_MB_OP_RD);
    mutex_lock(&qm->mailbox_lock);
    ret = qm_mb_nolock(qm, &mb);    // 从硬件读取数据到 mb
    mutex_unlock(&qm->mailbox_lock);
    // ...
    reg_pair_from_64(*base, &mb.base_h, &mb.base_l);  // BUG: *base(=0) 覆写了 mb 中的数据
    // mb 数据未被写回 *base
    return HISI_QM_DRVCRYPT_NO_ERR;
}

影响: qm_get_vft() 始终返回 base=0, num=1，依赖 VFT 正确配置的功能（多队列、SR-IOV）静默故障。

修复建议

替换为正确的数据赋值：

uint64_t val = ((uint64_t)mb.base_h << 32) | mb.base_l;
*base = val;
CR-03CVSS 7.4
hpre_rsa.c:771-807 — rsaes_pkcs_v1_5_decode()

PKCS#1 v1.5 填充验证存在可区分的错误路径

PKCS#1 v1.5 解填充函数对多个不同错误条件均返回 TEE_ERROR_BAD_PARAMETERS，但执行路径耗时不同：

if (em_len < PKCS_V1_5_MSG_MIN_LEN || em[0] != 0 || em[1] != ENCRYPT_PAD)
    return TEE_ERROR_BAD_PARAMETERS;  // 快速路径
// ...
if (i >= em_len)
    return TEE_ERROR_BAD_PARAMETERS;  // 中等路径
// ...
if (em_len - ps_len - PKCS_V1_5_FIXED_LEN > *out_len || ps_len < PKCS_V1_5_PS_MIN_LEN)
    return TEE_ERROR_BAD_PARAMETERS;  // 慢速路径

影响: 攻击者可通过时序区分错误类型，发动 Bleichenbacher 百万消息攻击 恢复 RSA 明文。

修复建议

替换为恒定时间验证块，将所有检查累加到一个状态标志中，仅返回一个通用错误码。

CR-04CVSS 5.9
hpre_rsa.c:895 — rsa_oaep_get_msg()

OAEP 标签比较使用非常恒定时间的 memcmp

OAEP 解码使用标准 memcmp 比较标签哈希：

if (memcmp(hash, db, lhash_len)) {
    EMSG("Hash 不相等");
    return TEE_ERROR_BAD_PARAMETERS;
}

影响: 产生逐字节的时序预言机，违反了 RSA 解密路径必须恒定时间的原则。

修复建议

替换为恒定时间比较：

int ct_neq = 0;
for (size_t i = 0; i < lhash_len; i++)
    ct_neq |= hash[i] ^ db[i];
if (ct_neq) {
    EMSG("Hash 不相等");
    return TEE_ERROR_BAD_PARAMETERS;
}
CR-05CVSS 5.9
hpre_montgomery.c:216-220

X25519/X448 坐标比较使用非常恒定时间的 memcmp

在蒙哥马利标量乘法之前，使用 memcmp 检查 u-坐标与域素数：

ret = memcmp(u + offset, p + offset, dsize);
if (ret >= 0) {
    EMSG("u >= p");
    return HISI_QM_DRVCRYPT_EINVAL;
}

影响: 逐字节泄露 u 与 p 的比较时序信息，可对 Diffie-Hellman 密钥交换进行时序攻击。

修复建议

使用恒定时间的无符号比较替代 memcmp。

sec_authenc.c:512-515

释放后将错误的指针置 NULL

if (ae_drv_ctx->dst.data) {
    free(ae_drv_ctx->dst.data);
    ae_drv_ctx->src.data = NULL;  // BUG: 应为 dst.data

void sec_authenc_copy_state(struct sec_aead_drv_state *dst,

HI-03
sec_cipher.h:68, sec_cipher.c 全文

IV 数组大小与长度类型不一致

struct sec_cipher_ctx {
    uint64_t iv[SEC_MAX_IV_SIZE];  // 2 × uint64_t = 16 字节
    uint8_t iv_len;                // 最大 255
    // ...
};

问题: iv_len 为 uint8_t（最大 255），但 iv 数组仅 16 字节。若验证被绕过，后续 memcpy(c_ctx->iv, iv, c_ctx->iv_len) 将溢出栈缓冲区。

影响: 栈缓冲区溢出，可能导致 TEE 中的内存损坏或代码执行。

修复建议

在每次赋值时验证 iv_len：iv_len > sizeof(c_ctx->iv)。

MD-01
hpre_dh.c:165, hpre_montgomery.c:133, hpre_ecc.c:889

hw_get_random_bytes 返回值未检查

多处调用 hw_get_random_bytes() 生成私钥、DH 指数和 ECDSA 随机数时未检查返回值。若硬件 RNG 失败或熵不足，密码操作将在弱随机性下继续执行。

修复建议

检查返回值并传播错误。对 ECDSA 签名建议增加软件 RNG 回退。

LW-01
hpre_rsa.c

RSA 栈变量未显式清零

RSA 操作中的栈分配缓冲区在函数返回前未使用 memzero_explicit 清除，密钥材料可能残留。

LW-02
hpre_montgomery.c

蒙哥马利私钥标量未擦除

私钥标量缓冲区通过 free() 释放前未用 memzero_explicit 擦除。

LW-03
sec_pbkdf2.c

PBKDF2 盐值和密码未清理

PBKDF2 消息结构中的 salt 和 base_key 缓冲区位于栈上，操作完成后未清零。

修复建议

操作结束时调用 memzero_explicit(msg->salt, msg->salt_len) 和 memzero_explicit(msg->base_key, msg->key_len)。

附加观察

代码质量

  • 释放并擦除抽象: HPRE 代码中 free_wipe() 使用一致，应推广到 SEC 代码。
  • 无释放后使用证据: 大多数位置遵循了分配/使用/释放后置 NULL 模式（HI-01 除外）。
  • 锁机制: hisi_qm.c 中互斥锁使用符合标准模式，无死锁竞争。

构建配置

  • 模块由 CFG_HISILICON_CRYPTO_DRIVER 和 CFG_HISILICON_ACC_V3 控制。
  • 密码算法通过标准 CFG_CRYPTO_* 变量管理。
